# Design — Meu Markdown v1.9.0

**Data**: 2026-05-20
**Autor**: Adriano Caldeira (com assistência do Claude)
**Status**: Aprovado para implementação

## Objetivo

Adicionar operações de arquivo no painel Explorer e corrigir o comportamento de abas órfãs ao trocar workspace.

Entregas:

1. **Novo arquivo `.md`** (Ctrl+N) no diretório selecionado.
2. **Nova pasta** (Ctrl+Shift+N).
3. **Copiar/Recortar/Colar** arquivos e pastas (Ctrl+C/X/V) via Windows clipboard.
4. **Drag-and-drop** entre diretórios no Explorer (move por padrão; Ctrl+drag copia).
5. **Fechar abas órfãs** ao trocar workspace — abas de arquivos fora do novo workspace root fecham automaticamente, abas dentro permanecem.

## Não-objetivos

- Multi-selection (1 item por operação).
- Lixeira do Windows (delete continua permanente).
- Undo de operações de arquivo.
- Operações via menu Arquivo no top (só pelo painel Explorer e atalhos).
- Atalhos do tipo "Duplicar" (use Copy + Paste).

## Arquitetura

```
ExplorerPanel (UI)
  ├── ContextMenu items + Click handlers (OnNewFile, OnNewFolder, OnCopy, OnCut, OnPaste)
  ├── InputBindings (Ctrl+N, Ctrl+Shift+N, Ctrl+C/X/V)
  ├── Drag-and-drop (PreviewMouseLeftButtonDown/Move, DragEnter/Over/Drop)
  └── delega lógica de FS para
       └── FileOperationsService (novo, testável)
             ├── CreateNewFile(dir, baseName, ext) → newPath
             ├── CreateNewFolder(dir, baseName) → newPath
             ├── CopyFile(source, destDir, onConflict) → destPath
             ├── MoveFile(source, destDir, onConflict) → destPath
             └── GenerateUniqueName(dir, baseName, ext?) → "novo-arquivo (2).md"

MainViewModel.OnWorkspaceChanged (novo)
  └── itera Tabs; fecha as cujo FilePath não está sob o novo RootPath
```

## Componentes

### 1. `FileOperationsService` (novo)

Serviço **stateless** com métodos puros de operação de FS. Sem dependência de UI — testável com xUnit + temp dirs.

```csharp
namespace MeuMarkdown.Services;

public class FileOperationsService
{
    public string CreateNewFile(string directory, string baseName, string extension);
    public string CreateNewFolder(string directory, string baseName);
    public string CopyFile(string sourcePath, string destDirectory, Func<string, bool>? onConflict = null);
    public string MoveFile(string sourcePath, string destDirectory, Func<string, bool>? onConflict = null);

    // Internal/static helper exposto para teste:
    internal static string GenerateUniqueName(string directory, string baseName, string? extension);
}

public class FileOperationException : Exception { /* mensagens user-friendly */ }
```

Regras de geração de nome único:

- Sem conflito → retorna `baseName + extension` (ex: `novo-arquivo.md`)
- Com conflito → adiciona ` (2)`, ` (3)`, etc, **antes da extensão** (ex: `novo-arquivo (2).md`)
- Para pasta sem extensão: `nova-pasta`, `nova-pasta (2)`, ...

`onConflict`: callback (`string existingPath → bool overwrite`). Quando o destino existe:
- `null` ou retorna `false` → cancela operação, retorna path original sem mudança (ou lança exceção? Decisão: **retorna `null` quando cancelado**).
- Retorna `true` → sobrescreve.

Para copy/move dentro do mesmo dir, **sem callback** (já gera nome único automaticamente).

### 3. Clipboard

Usa Windows `DataFormats.FileDrop` (lista de paths) + `DataFormats.Text` (path único como fallback) + um flag adicional:

```csharp
private const string CutFlagFormat = "Preferred DropEffect";

public void CopyToClipboard(string path)
{
    var data = new DataObject(DataFormats.FileDrop, new[] { path });
    data.SetData(DataFormats.Text, path);
    Clipboard.SetDataObject(data);
}

public void CutToClipboard(string path)
{
    var data = new DataObject(DataFormats.FileDrop, new[] { path });
    data.SetData(DataFormats.Text, path);
    // Flag pra Move (compatível com Windows Explorer)
    var flag = new byte[] { 2, 0, 0, 0 }; // DROPEFFECT_MOVE
    data.SetData(CutFlagFormat, new MemoryStream(flag));
    Clipboard.SetDataObject(data);
}
```

**Paste** lê `FileDrop` + verifica flag `Preferred DropEffect`. Se valor = `2` → Move; senão → Copy.

**Vantagem dessa abordagem**: copiar do Windows Explorer e colar no Meu Markdown funciona. E vice-versa.

### 4. Drag-and-drop

```csharp
private Point _dragStartPoint;
private FileNode? _dragSourceNode;

private void OnTreePreviewMouseLeftButtonDown(object sender, MouseButtonEventArgs e)
{
    _dragStartPoint = e.GetPosition(null);
    _dragSourceNode = FindNode(e.OriginalSource);
}

private void OnTreePreviewMouseMove(object sender, MouseEventArgs e)
{
    if (e.LeftButton != MouseButtonState.Pressed) return;
    if (_dragSourceNode == null) return;

    var diff = _dragStartPoint - e.GetPosition(null);
    if (Math.Abs(diff.X) < SystemParameters.MinimumHorizontalDragDistance &&
        Math.Abs(diff.Y) < SystemParameters.MinimumVerticalDragDistance) return;

    var data = new DataObject(DataFormats.FileDrop, new[] { _dragSourceNode.FullPath });
    DragDrop.DoDragDrop(FileTree, data,
        Keyboard.Modifiers == ModifierKeys.Control ? DragDropEffects.Copy : DragDropEffects.Move);
    _dragSourceNode = null;
}

private void OnTreeDragOver(object sender, DragEventArgs e)
{
    e.Effects = DragDropEffects.None;
    var target = FindNode(e.OriginalSource);
    if (target?.IsDirectory != true) return;
    if (!e.Data.GetDataPresent(DataFormats.FileDrop)) return;

    e.Effects = (e.KeyStates & DragDropKeyStates.ControlKey) != 0
        ? DragDropEffects.Copy
        : DragDropEffects.Move;
    e.Handled = true;
}

private void OnTreeDrop(object sender, DragEventArgs e)
{
    var target = FindNode(e.OriginalSource);
    if (target?.IsDirectory != true) return;
    var sources = (string[])e.Data.GetData(DataFormats.FileDrop);
    if (sources?.Length != 1) return;

    var isCopy = (e.KeyStates & DragDropKeyStates.ControlKey) != 0;
    // Cancela drop em si mesmo
    if (Path.GetDirectoryName(sources[0]) == target.FullPath) return;
    if (sources[0] == target.FullPath) return;

    try
    {
        if (isCopy) _fileOps.CopyFile(sources[0], target.FullPath, AskOverwrite);
        else _fileOps.MoveFile(sources[0], target.FullPath, AskOverwrite);
    }
    catch (Exception ex)
    {
        MessageBox.Show($"Erro ao mover/copiar:\n{ex.Message}", "Erro", MessageBoxButton.OK, MessageBoxImage.Error);
    }
}
```

`FindNode(originalSource)` percorre o visual tree até achar um `TreeViewItem` e retorna seu `DataContext as FileNode`.

### 5. Fechar abas órfãs no trocar workspace

**Novo método em `MainViewModel`:**

```csharp
public void CloseTabsOutsideWorkspace(string? newWorkspaceRoot)
{
    if (string.IsNullOrEmpty(newWorkspaceRoot)) return;

    var tabsToClose = Tabs
        .Where(t => !IsUnderRoot(t.FilePath, newWorkspaceRoot))
        .ToList();

    foreach (var tab in tabsToClose)
    {
        if (!TryCloseTab(tab)) break; // cancelado pelo user em prompt de save
    }
}

private static bool IsUnderRoot(string filePath, string root)
{
    var normalizedFile = Path.GetFullPath(filePath);
    var normalizedRoot = Path.GetFullPath(root).TrimEnd(Path.DirectorySeparatorChar)
        + Path.DirectorySeparatorChar;
    return normalizedFile.StartsWith(normalizedRoot, StringComparison.OrdinalIgnoreCase);
}
```

`TryCloseTab` existente já lida com abas dirty (prompt salvar/descartar/cancelar).

**Chamada do método** em `MainWindow.xaml.cs`, dentro do `OnOpenFolderRequested` (dialog handler), **antes** de chamar `_viewModel.WorkspaceService.Open(newPath, ...)`. Razão: se algumas abas pediram cancel, podemos preferir ainda assim trocar o workspace, mas as abas que o user cancelou ficam órfãs. Decisão: **trocar workspace primeiro, depois fechar órfãs** — mais previsível pro user.

Ordem final:
1. User confirma novo path no dialog
2. `_viewModel.WorkspaceService.Open(newPath, showAllFiles)`
3. `_viewModel.CloseTabsOutsideWorkspace(newPath)` — fecha as órfãs
4. UI reage à mudança de Root

## Segurança

- `FileOperationsService` valida que paths não escapam do workspace via `..` antes de qualquer operação destrutiva.
- Delete continua exigindo confirmação (já existe).
- Move/Copy com conflito sempre pergunta antes de sobrescrever (exceto colar no mesmo dir, onde gera nome único).

## Edge cases

| Caso | Comportamento |
|---|---|
| `Ctrl+N` sem workspace aberto | No-op silencioso (sem feedback de erro, igual aos atalhos existentes) |
| `Ctrl+V` sem clipboard contendo `FileDrop` | No-op |
| `Ctrl+V` com vários paths no clipboard (vindo do Explorer) | Processa apenas o primeiro (YAGNI multi-select) |
| Renomear pra nome existente | MessageBox de erro (já existe) |
| Drop em si mesmo | Cancela silenciosamente |
| Drop num arquivo (não pasta) | Cancela silenciosamente |
| Mover/copiar de fora do workspace pra dentro | Permitido (mesmo padrão do Windows Explorer) |
| Trocar workspace com 5 abas, 3 fora do novo root, 1 dirty | Fecha as 2 limpas; pergunta save da dirty; se user cancela, ela fica órfã |

## Riscos

| Risco | Mitigação |
|---|---|
| Clipboard `FileDrop` cross-process pode quebrar em sandboxed scenarios | Validar com try/catch + fallback pra `DataFormats.Text` |
| Drag-and-drop pode disparar drop em multiplos handlers se nested | Usar `e.Handled = true` no Drop |
| Move via Windows Explorer pode falhar entre drives diferentes (não-atomic) | `File.Move` lança IOException; capturar e mostrar erro user-friendly |
| Refresh do TreeView atrasa (FileSystemWatcher debounce 800ms) | Forçar `WorkspaceService.Refresh()` após operações |
| Trocar workspace com aba dirty cancelada | Aceitar comportamento "órfã permanece" — documentar |
