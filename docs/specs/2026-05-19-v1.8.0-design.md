## Objetivo

Lançar a release **v1.8.0** com três entregas de produto que fecham gaps relevantes em relação aos concorrentes (Obsidian, Typora, Markdown Monster) e documentam uma feature já implementada mas não anunciada:

2. **Diagramas Mermaid e fórmulas KaTeX** no preview, com suporte ao export PDF/HTML (gap Ferrite/Obsidian).

## Componentes em detalhe

### Wiki-links

#### Sintaxe

| Forma | Comportamento |
|---|---|
| `[[Arquivo]]` | Resolve `Arquivo.md` no workspace; exibe `Arquivo` |
| `[[Arquivo\|Texto]]` | Resolve `Arquivo.md`; exibe `Texto` |
| `[[Arquivo#heading]]` | Resolve + ancora `#heading` no link gerado |
| `[[Arquivo#heading\|Texto]]` | Combinação completa |
| `[[Pasta/Arquivo]]` ou `[[Pasta/Arquivo.md]]` | Path relativo ao diretório do arquivo atual (não busca por nome no workspace). `.md` é anexado se ausente. |

#### Resolução de nome (`WorkspaceService.ResolveWikiLink`)

```csharp
public record WikiLinkResolution(string AbsolutePath, string DisplayName);

public WikiLinkResolution? ResolveWikiLink(string target, string? currentFileDir);
```

Algoritmo:

1. Se `target` contém `/` ou `\` → tratar como path relativo a `currentFileDir`. Se não terminar em `.md`, anexar `.md`. Verificar existência via `File.Exists`. Existe → resolve; não → `null`.
2. Senão (nome simples):
   - Normalizar: `key = target.ToLowerInvariant()` (sem extensão).
   - Lookup no índice `Dictionary<string, List<string>>` (key → paths absolutos).
   - 0 matches → retorna `null` (broken).
   - 1 match → resolve.
   - N matches → **ranking determinístico** por:
     1. **Distância LCA** entre cada candidato e `currentFileDir`. Definida como `(hops do candidato até o LCA) + (hops do currentFileDir até o LCA)`. Menor vence.
     2. Empate em distância → ordem alfabética ascendente do path absoluto (case-insensitive).
   - Exemplo: `currentFileDir = /workspace/area-a/sub`, candidatos `/workspace/area-a/Foo.md` (dist 1+0=1) e `/workspace/area-b/Foo.md` (dist 1+2=3) → vence o primeiro.

#### Cache do índice

- Estrutura: `Dictionary<string, List<string>>` no `WorkspaceService` (lazy build na primeira resolução).
- Invalidação: o `WorkspaceService` já possui `FileSystemWatcher` para a árvore do workspace; adicionar handler que invalida o índice quando arquivos `.md` são criados/renomeados/deletados.
- Sem workspace aberto → resolver retorna `null` sempre, e os wiki-links são renderizados como **broken**.

#### Renderização HTML

| Caso | Output |
|---|---|
| Resolvido | `<a href="mdnav://open?path={encoded}{#heading}" class="wikilink">{displayText}</a>` |
| Broken | `<span class="wikilink-broken" title="Arquivo não encontrado">{rawText}</span>` |

O `mdnav://` é reescrito automaticamente — não é necessário passar pelo regex `RewriteRelativeLinks` (o renderer já emite no formato final).

**Bug pré-existente que entra no escopo** (necessário para `[[Arquivo#heading]]` funcionar):

Hoje, links comuns como `[texto](arquivo.md#heading)` são reescritos para `mdnav://open?path=arquivo.md%23heading`. No clique, [`MarkdownPreview.xaml.cs:264-267`](../../src/MeuMarkdown/Controls/MarkdownPreview.xaml.cs#L264) decodifica e dispara `LinkClicked("arquivo.md#heading")`. Em [`MainWindow.xaml.cs:481`](../../src/MeuMarkdown/MainWindow.xaml.cs#L481), `NavigateTo` chama `FileService.ResolvePath` → `File.Exists("…/arquivo.md#heading")` falha → navegação silenciosamente abortada.

Correção incluída no escopo:

- `LinkClicked` passa a expor um segundo parâmetro opcional `fragment` (heading sem `#`), evento muda para `Action<string, string?>?`.
- No emit do `mdnav://`, separar `path` e `fragment` em query params distintos: `mdnav://open?path={encoded}&fragment={encoded}`.
- `NavigationService.NavigateTo(filePath, currentDirectory, fragment?)` ganha o parâmetro. Quando o tab de destino abre, o `MainViewModel` dispara um scroll-to-heading via `Outline.ScrollToHeading(fragment)` (reusar lógica existente do Outline panel).
- Aplicável tanto a wiki-links quanto a links markdown comuns `[x](y.md#z)`.

#### Autocomplete `[[`

- Trigger: handler `TextEntered` no `TextEditor` do AvalonEdit. Quando o caractere digitado é `[` e o caractere anterior também é `[`.
- Guarda: ignora dentro de fenced code block (heurística — contar tripletas ` ``` ` antes do caret; se ímpar, está em code).
- Janela: `CompletionWindow` com `ICompletionData` por arquivo `.md` do workspace, ordenado por fuzzy match contra o que o user digitou após o `[[`.
- Limite: 50 itens.
- Inserção (Tab/Enter): completa com `Nome]]` e move o caret para depois do `]]`.
- Aliases: Esc fecha sem inserir.

### Mermaid + KaTeX

#### Markdig

- Adicionar `.UseMathematics()` no pipeline em `MarkdownService` ctor — reconhece `$..$` e `$$..$$`, emite `<span class="math">` / `<div class="math">`.
- Para Mermaid, **não é necessária extensão Markdig**: blocos fenced com linguagem `mermaid` já são emitidos como `<pre><code class="language-mermaid">…</code></pre>`. O JS do template detecta e substitui pelo elemento que o Mermaid precisa.

Sobre fontes WOFF2: KaTeX referencia fontes via URL relativa no CSS. Duas opções:

- **A. Virtual host mapping** — registrar `local.katex/` apontando pra um diretório temporário onde extraímos as fontes no startup. Funciona, mas adiciona I/O.
- **B. Inline data URLs no CSS** — reescrever o `katex.min.css` para inlinear as fontes em `url("data:font/woff2;base64,…")`. CSS cresce ~25KB → ~280KB. Sem I/O, zero setup.

**Escolha: B**, processada em build-time via script PowerShell (`scripts/inline-katex-fonts.ps1`) que produz `katex.min.css` (embedded resource) com fonts inlined. Documentado no script. Re-rodar quando bumpar versão do KaTeX.

#### `renderEnhancements()` no preview-template

```js
async function renderEnhancements() {
  // Mermaid: <pre><code class="language-mermaid">…</code></pre> → <div class="mermaid">…</div>
  document.querySelectorAll('pre code.language-mermaid').forEach(el => {
    const div = document.createElement('div');
    div.className = 'mermaid';
    div.textContent = el.textContent;
    el.closest('pre').replaceWith(div);
  });
  if (window.mermaid) {
    try { await mermaid.run({ querySelector: '.mermaid' }); }
    catch (e) { /* mermaid já desenha box de erro inline */ }
  }

  if (window.renderMathInElement) {
    renderMathInElement(document.body, {
      delimiters: [
        { left: '$$', right: '$$', display: true },
        { left: '$',  right: '$',  display: false }
      ],
      throwOnError: false
    });
  }
}
```

- `updateContent(html)` (existente) passa a `await renderEnhancements()` no final.
- `setTheme(dark)` (existente) passa a chamar `mermaid.initialize({ startOnLoad: false, theme: dark ? 'dark' : 'default' })` e re-renderizar os `.mermaid` existentes.
- Bootstrapping: `mermaid.initialize({ startOnLoad: false, theme: 'default' })` é chamado uma vez no `<script>` inicial do template.

#### PrintToPdfAsync — suporte a render async

Modificação em `MarkdownPreview.PrintToPdfAsync`:

```csharp
webView.CoreWebView2.Navigate("file:///" + sourceHtmlPath...);
await tcs.Task; // NavigationCompleted
await webView.CoreWebView2.ExecuteScriptAsync("(async () => { await renderEnhancements(); return 'done'; })()");
var success = await webView.CoreWebView2.PrintToPdfAsync(destPdfPath, printSettings);
```

`ExecuteScriptAsync` espera o promise do IIFE resolver — o PDF só é gerado depois dos diagramas/fórmulas renderizarem.

## Edge cases

| Caso | Comportamento |
|---|---|
| Sem workspace aberto | Wiki-links renderizados como broken |
| `[[Foo]]` dentro de `` `code inline` `` | Markdig não invoca inline parser em code → literal |
| `[[Foo]]` dentro de fenced code | Idem — literal |
| Wiki-link com pipe dentro do alias | Pipe é separador único; aliases não podem conter pipe (limitação aceitável documentada) |
| Múltiplos `.md` com mesmo nome | Resolve o mais próximo do current dir (menor distância vertical); empate → alfabético |
| Mermaid sintaxe inválida | Mermaid renderiza painel de erro inline |
| KaTeX sintaxe inválida | KaTeX exibe a string em vermelho com tooltip |
| Troca de tema com diagramas renderizados | `setTheme` re-inicializa Mermaid e re-renderiza |
| Export PDF de doc com Mermaid/KaTeX | `ExecuteScriptAsync("await renderEnhancements()")` antes do `PrintToPdfAsync` |
| Doc grande com 50+ diagramas | `mermaid.run()` é sequencial; aceitável (alguns segundos de delay) |
| Workspace muito grande (10k+ arquivos) | Índice é lazy + cacheado; rebuild ocorre só em mudanças. Aceitável. |

## Riscos

| Risco | Mitigação |
|---|---|
| Tamanho do binário cresce ~1MB | Aceitável (~165MB → ~166MB) |
| `mermaid.run()` lento em docs com muitos diagramas | Async, não bloqueia editor |
| Mudança no Markdig pipeline quebra layout existente | Plano de testes manuais cobre regressão |
| Resolução de wiki-links ambígua em workspaces grandes | Algoritmo documentado e determinístico (distância + alfabético) |
| `EmbeddedResource` falha de carregamento | Logs em caso de exceção; carrega no ctor do `MarkdownService` |
| KaTeX font inlining script desatualiza ao bumpar KaTeX | Script documentado em `scripts/inline-katex-fonts.ps1` com instruções |
